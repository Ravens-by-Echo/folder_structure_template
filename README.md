# Folder Structure Template

Folder Structure Developed with a focus on:

- C / C++ (Embedded)
- Python
- Bash

But if can be used and fitted for other programming languages.

**A structured layout improves:**

- **Maintainability** – developers can quickly find relevant code

- **Scalability** – new modules, tools, or systems can be added without clutter

- **Reusability** – reusable components remain independent from runtime logic

- **Separation of concerns** – configuration, implementation, and testing are clearly separated

- **Onboarding** – new contributors understand the project faster

This repository serves as a template for a structured and easy to maintain project. It uses a numbered folder structure to keep components organized, scalable, and easy to navigate. Each directory has a clear purpose and separates reusable code, configuration, runtime implementations, and supporting tools.

## Directories Overview

### .environment

Contains environment configurations, a template of this can be found in [mini-forge_environment_template](https://github.com/Ravens-by-Echo/mini-forge_environment_template).

### 01_documentation

Contains project documentation such as architecture descriptions, design decisions, diagrams, and user or developer guides.

Purpose:

- Explain how the system works
- Document interfaces, modules, and workflows
- Provide reference material for developers and users

### 02_configuration

Stores configuration files used by the project.

Examples:

- environment settings
- device configurations
- calibration parameters
- runtime settings

Keeping configuration separate allows behavior to be adjusted without modifying source code.

### 03_modules

Contains reusable modules, libraries, and core components.

Examples:

- peripherals behavior
- communication interfaces
- algorithms
- hardware abstraction

Modules should be self-contained and reusable across different runtime implementations.

### 04_external

Stores third-party dependencies and external libraries used by the project.

Examples:

- vendor SDKs
- open-source libraries
- external drivers

Separating these from internal modules avoids mixing project code with external code.

### 05_tests

Contains tests that validate the functionality of modules and runtime implementations.

Examples:

- unit tests
- integration tests
- simulation tests

Testing ensures that changes do not break existing functionality and improves system reliability.

### 06_tools

Contains development and maintenance tools used to support the project.

Examples:

- Python utilities
- Bash automation scripts
- data processing scripts
- build helpers

These tools assist development but are not part of the runtime system.

### 07_runtime_implementation

Contains the actual runtime systems or applications built using the modules.

Examples:

- firmware implementations
- system controllers
- monitoring programs
- simulation environments

This directory represents the final assembled behavior of the system, combining modules and configuration.

## Runtime and autogenerated Folders

Folder such as build, which are generated at compilation should not be included in the folder structure since it can contain environment specific configuration or private information.

Autogenerated folders should also not be included since it will only occupy unnecessary space in the project.

These all should be added to the `.gitignore` files
